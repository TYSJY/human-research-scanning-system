# Research OS 0.4.4 产品化可用性重构交付说明

这次重构的目标不是继续增强“工程内核感”，而是在 **保留现有 runtime / state / orchestration 能力** 的前提下，把系统升级成 **普通用户第一次打开也不害怕、第一次运行也知道在做什么、第一次报错也知道怎么修** 的产品化系统。

本次重构遵循了 4 个硬约束：

1. **不脱离现有项目骨架另起炉灶**：保留 `research_os/`、`control_plane/`、`templates/`、`projects/` 的主结构。
2. **不只改文案**：直接重构了 CLI 默认入口、onboarding、doctor、demo、quickstart、status、approve、audit，并新增可执行的 Web UI。
3. **不只做术语包装**：把普通用户主路径从底层 runtime 术语里抽出来，用新入口和新流程承接。
4. **不让普通用户先学 runtime 才能开始**：高频主流程已经变成“看状态 → 继续一步 → 批准下一项 → 看健康检查 → 生成审计”。

---

## 1. 普通用户视角的问题诊断

### 1.1 当前系统原本最不友好的地方

原项目的底层能力已经很完整，但它更像“工程团队内部可操控的 runtime”，而不是“第一次打开就知道怎么开始的产品”。对普通用户来说，最主要的问题不是功能不够，而是 **入口方式、信息组织和术语暴露方式** 不友好。

最明显的阻碍有 6 类：

1. **默认入口过硬**
   - 用户一进来面对的是大量 CLI 子命令。
   - 默认路径要求先理解项目路径、provider、orchestrate、workloop 等概念。
   - 没有“先玩一下，再理解”的体验。

2. **术语负担过大**
   - `run / gate / evaluation / artifact / selector / scheduler / worker` 对普通用户并不直观。
   - 这些概念本身需要保留，但不应该在第一次使用时成为前置知识。

3. **没有足够短的 happy path**
   - 原系统可以做很多事，但用户不容易知道“我第一步做什么最合适”。
   - 缺少 `quickstart → 看状态 → 跑一步 → 成功/失败都有下一步` 这样的闭环。

4. **CLI 作为唯一入口，心理门槛太高**
   - 即使 CLI 技术上合理，普通用户仍然会天然畏惧长命令、参数和 JSON 输出。
   - 缺少一个直接能看懂项目状态、点击常见操作的主入口。

5. **错误提示偏技术检查，不够用户修复导向**
   - 原有校验信息对工程师有帮助，但不总是能直接回答“我现在该怎么修”。
   - 普通用户需要的是“为什么会这样”“是否严重”“先点哪里”。

6. **成功后的 next step 不够集中**
   - 系统状态散落在多个 registry、report、task、run、gate 中。
   - 用户第一次成功后，仍不容易知道下一步优先级。

### 1.2 当前 CLI 最容易让普通用户产生畏惧感的点

- 子命令密度高，初看像内部控制台而不是用户产品。
- 路径和参数感太强，需要先理解目录布局。
- 默认命令返回偏结构化信息，不够“人话”。
- 一些高级命令看起来像第一次使用就必须掌握。
- 没有明显区分“新手入口”和“高级入口”。

### 1.3 哪些概念必须保留，但不该直接暴露

这些概念是系统核心能力的一部分，必须保留，但不应作为普通用户第一层语言：

- `stage machine`
- `task graph`
- `gate`
- `run`
- `worker`
- `evaluation`
- `artifact`
- `scheduler`
- `selector`

本次重构的做法不是删除它们，而是：

- **普通用户界面先显示人话标签**
- **高级信息放到折叠区或高级命令中**
- **需要时再向下映射到底层术语**

## 2. 产品可用性评审

### 2.1 首次打开体验

**重构前**：像一个可编排 runtime，用户需要主动理解系统。

**重构后**：默认 `ros` 会直接进入欢迎页，告诉用户：

- 这是什么
- 最短体验路径是什么
- 最常用命令是什么
- 已发现的项目是什么
- 下一步推荐是什么

### 2.2 onboarding 清晰度

**重构前**：需要靠 README、历史文档和理解命令语义来拼装首条路径。

**重构后**：形成了 4 层 onboarding：

1. `ros` 默认欢迎页
2. `ros quickstart` 一键体验路径
3. `START_HERE.md` 项目内首读文件
4. `ros ui` 浏览器工作台

### 2.3 工程师术语负担

**重构前**：底层术语直接暴露在主流程中。

**重构后**：默认展示改为：

- 项目阶段
- 待办清单
- 人工确认
- 一次实验 / 一次任务
- 结果检查
- 输出文件
- 下一步建议

高级术语仍然保留，但不再占据主视图。

### 2.4 默认路径长度

**重构前**：首次成功路径偏长，需要理解 orchestration、provider 和项目细节。

**重构后**：普通用户 5~10 分钟 happy path 可收敛为：

1. `ros quickstart`
2. `ros ui`
3. `ros run <项目路径>`
4. `ros approve <项目路径>`（如需要）
5. `ros doctor <项目路径>`（遇到问题时）

### 2.5 错误修复可读性

**重构前**：更偏系统校验描述。

**重构后**：`doctor` 和 `validate` 都转成“标题 + 原因 + 修复建议”的形式，重点回答：

- 这是阻塞问题还是提醒
- 为什么会这样
- 你现在先做什么

### 2.6 UI 是否真正承担主入口角色

**重构前**：没有。

**重构后**：Web UI 不是装饰层，而是能直接执行：

- 创建空项目
- 复制演示项目
- 继续推进一步 / 多步
- 批准下一项
- 生成审计报告
- 健康检查
- 查看项目状态、任务、注意事项、最近 runs / evaluations / artifacts / sessions

## 3. P0 / P1 / P2 可用性问题清单

### P0（必须先解决，否则普通用户几乎无法顺利起步）

1. **没有新手主入口**
2. **首次启动不知道系统是什么、能做什么、下一步做什么**
3. **缺少一键体验路径**
4. **没有 UI 主入口，CLI 成为唯一入口**
5. **错误信息不够修复导向**
6. **成功后没有明显的 next step 推荐机制**

### P1（会显著降低体验，虽非绝对阻塞）

1. 帮助信息更像框架说明，不像产品说明
2. 演示项目入口不够集中
3. 默认状态展示过于技术化
4. 同类样例项目容易混淆
5. 项目内缺少更明确的“从这里开始”文件

### P2（提升整体完成度与产品感）

1. UI 还可以继续加入更多可视化趋势与更细粒度操作
2. 可以继续细化“新手模式 / 高级模式”的分层呈现
3. 可以加入更强的首次成功庆祝、进度里程碑与推荐动作
4. 可以补更多针对真实 provider 的引导与配置助手

## 4. 以 onboarding + Web UI 为核心的重构策略

本次采用的是 **“双层入口 + 统一 happy path + 术语降噪”** 的重构策略。

### 4.1 双层入口

- **普通用户主入口**：`quickstart / init / demo / status / doctor / run / approve / audit / ui`
- **高级用户入口**：保留 `plan / scheduler / orchestrate / workloop / create-run / run-worker ...`

这意味着 CLI 没被废掉，但它被重新分层了。

### 4.2 Happy path 固化

把默认成功路径从“自己理解底层流程”改成“系统给你下一步”：

- `ros quickstart`：一键准备 demo
- `ros init`：创建空项目
- `ros status`：告诉你当前状态
- `ros run`：自动推进下一步
- `ros approve`：处理待确认项
- `ros doctor`：解释为什么卡住
- `ros audit`：输出审计结果

### 4.3 术语隐藏而非删除

底层对象仍保持原结构和语义，但普通用户默认看到：

- 项目阶段 ← `stage`
- 待办清单 ← `task graph`
- 人工确认 ← `gate`
- 一次实验 / 一次任务 ← `run`
- 本地执行器 ← `worker`
- 结果检查 ← `evaluation`
- 输出文件 ← `artifact`
- 任务排队器 ← `scheduler`
- 优先结果选择器 ← `selector`

### 4.4 可视化入口最小可行版本（MVP）

优先做 **轻量、直接运行、零外部依赖的 Web UI**，因为：

- 实现速度快
- 普通用户可理解性更好
- 比 TUI 更容易承担主入口角色
- 不需要引入额外前端工程体系

## 7. 新手入口设计说明

### 7.1 新手主入口的设计原则

新手入口必须回答 5 个问题：

1. 这是什么
2. 我现在能做什么
3. 我下一步该点什么 / 执行什么
4. 出错后该怎么修
5. 成功后下一步是什么

### 7.2 具体落地

#### 默认 `ros`

展示欢迎页、最短路径、常用命令、已发现项目和下一步建议。

#### `ros quickstart`

默认复制内置演示项目，缩短首次成功路径。

#### `ros init`

创建空项目，并输出项目概况与下一步建议。

#### `ros status`

把原本偏技术的项目状态，重写成普通用户能读懂的项目摘要。

#### `ros doctor`

把问题诊断从“技术检查结果”重构成“是否阻塞 + 原因 + 修复建议”。

## 8. onboarding 设计说明

### 8.1 Onboarding 目标

让用户在第一次接触的 5~10 分钟内完成一次成功体验，并建立正确心智：

- 系统会保存状态
- 系统会提示下一步
- 遇到阻塞时系统会解释
- UI 是主要入口，CLI 是可选但强大的补充入口

### 8.2 Onboarding 流程

#### 路径 A：想先体验

1. `ros quickstart`
2. `ros ui`
3. 看项目概况
4. 点“继续一步”或执行 `ros run`
5. 遇到待确认时点“批准下一项”

#### 路径 B：想从空白开始

1. `ros init`
2. 阅读项目内 `START_HERE.md`
3. `ros ui <项目路径>`
4. `ros run <项目路径>`
5. `ros doctor <项目路径>`

### 8.3 成功后下一步推荐

系统会根据项目状态自动生成下一步建议，例如：

- 继续推进一步
- 批准某个等待确认的节点
- 查看健康检查
- 生成审计报告
- 打开 UI 查看最近任务和输出

## 9. Web UI 设计说明

### 9.1 为什么优先 Web UI 而不是 TUI

本轮优先级是“普通用户可用性”，因此优先实现：

- 更低理解门槛
- 更好状态总览
- 更明确按钮式操作
- 更少终端依赖

### 9.2 Web UI MVP 覆盖的核心路径

已覆盖：

- 查看项目概况
- 查看当前阶段 / 目标 / 健康度
- 查看待处理事项
- 查看最近 runs / evaluations / artifacts / sessions
- 查看下一步建议
- 启动常用操作：
  - 创建空项目
  - 复制演示项目
  - 继续一步
  - 连续推进
  - 批准下一项
  - 直接批准 gate / run
  - 健康检查
  - 生成审计报告

### 9.3 设计取舍

- 不引入额外前端构建工具
- 不要求花哨视觉
- 优先“能操作、能解释、能推进”
- 高级术语放入折叠区，避免主界面压迫感

## 10. 术语隐藏 / 重命名方案

| 底层术语 | 对外叫法 | 说明 |
|---|---|---|
| stage | 项目阶段 | 项目当前处于哪一步 |
| task graph | 待办清单 | 系统维护的任务列表 |
| gate | 人工确认 | 关键节点需要人工点头 |
| run | 一次实验 / 一次任务 | 一次具体执行与记录 |
| worker | 本地执行器 | 真正去执行任务的进程 |
| evaluation | 结果检查 | 判断结果是否完整可信 |
| artifact | 输出文件 | 配置、日志、表格、结果等 |
| scheduler | 任务排队器 | 决定哪些任务先开始 |
| selector | 优先结果选择器 | 帮助从多个结果里选更好的 |

策略不是“彻底改名”，而是：

- UI/新手 CLI 默认说人话
- 高级信息保留原术语
- 两套说法之间有明确映射

## 11. 新增的人性化能力说明

本轮新增的人性化能力包括：

1. **默认欢迎页**：普通用户第一次打开就知道这是干什么的。
2. **一键体验路径**：`quickstart` 默认落到 demo。
3. **项目内首读文件**：`START_HERE.md` 自动生成。
4. **项目概况页**：`status` 不再只像状态机输出。
5. **下一步推荐机制**：根据当前阶段、待确认项、健康度自动生成建议。
6. **错误修复建议**：校验结果与异常信息被翻译成可执行修复动作。
7. **可操作 Web UI**：不是静态 dashboard，而是能直接推进工作流。
8. **术语降噪**：普通用户不必先理解 runtime 词汇。
9. **样例去混淆**：演示样例与迁移样例已明确区分。

## 12. Breaking changes

本轮不是推倒式改版，但仍有一些行为变化需要注意：

1. `ros` 无参数时，不再直接表现为传统技术 CLI，而是进入欢迎页。
2. 新手推荐命令已经切换到 `quickstart / init / demo / status / doctor / run / approve / audit / ui`。
3. `summary` 现在等价于 `status`。
4. `init-project` 现在等价于 `init`。
5. `audit-report` 现在等价于 `audit`。
6. `validate` 的输出现在更偏人类可读，而不是只输出原始错误列表。
7. 首页项目展示会更偏产品化摘要，而不是原始底层状态。

高级命令没有被删除，只是从主入口退居二线。

## 13. 迁移说明

### 对已有项目

- 原有项目状态结构继续兼容。
- 核心 runtime / state / orchestration 没有推翻。
- 可以直接对现有项目使用新的 `status / doctor / run / approve / ui`。

### 对已有使用者

- 原高级命令仍可继续使用。
- 如果你之前习惯：
  - `summary` → 现在优先用 `status`
  - `init-project` → 现在优先用 `init`
  - `audit-report` → 现在优先用 `audit`

### 对 V4.1 / V3 项目

- `upgrade-v4_1` 与 `migrate-v3` 仍然可用。
- 迁移后可直接使用新 UI 和新手命令层。

## 结论

这次 0.4.4 的价值不在于“又新增了多少底层 capability”，而在于：

- **把已有 capability 重新组织成普通用户可进入、可理解、可推进的产品**
- **让 CLI 从唯一入口变成双层入口中的高级入口**
- **给系统补上 onboarding、下一步建议、修复建议和可操作 UI**

结果是：

- 普通用户第一次打开不那么害怕
- 普通用户第一次运行知道自己在做什么
- 普通用户第一次报错时知道怎么修
- 普通用户第一次成功后知道下一步是什么
